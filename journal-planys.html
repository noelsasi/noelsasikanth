Full story

My first real codebase. Not a tutorial project, not a hackathon, not a toy app with fake data. A startup building software for ROVs — Remotely Operated Vehicles — that go underwater to inspect pipelines, ship hulls, and dams. Real hardware. Remote locations. Clients who flew in from offshore oil platforms to see deliverables. And they handed me an entire module to own as an intern. That kind of trust changes how you work.

The Platform

I was brought in to help build WebDesk — an internal MVC web platform for managing complete underwater inspection campaigns from start to finish. By the time I left, it had 35 database models, 11 controller subsystems, and over 400 routes. The data hierarchy alone was a masterclass in domain modelling: Report → Jobs → Days → Dives → Observations → Defects. Every concept mapped to a real thing that happened underwater. You had to understand the physical process to model it right.

I built this on Node.js with Express, PostgreSQL for the database, Sequelize as the ORM, and Pug for templating. Authentication via Passport.js. This was before I'd touched React — it was server-rendered HTML, and it was exactly the right place to learn how a web server actually works: routing, middleware, session management, database transactions, rendering.

The Offline Problem

One of the more interesting engineering problems: ROV inspections happen in places with no internet. Offshore platforms, dam inspection sites, remote harbours. The report generator couldn't be a web app that called an API — it needed to run standalone.

The solution was Electron.js. I packaged the report UI into a Windows executable — an installable .exe that engineers could copy onto a laptop, take to site, and run without ever touching a browser or needing connectivity. This was my first encounter with the gap between "it works on my machine" and "it runs in the field." The distribution model matters as much as the software.

Async Video Processing

ROV dives are recorded on video. Processing those videos for the reports — trimming, compressing, extracting frames — is CPU-heavy and takes time. Doing it synchronously in a web request would have made the UI hang for minutes.

I implemented async job processing using bee-queue, a Redis-backed job queue for Node.js. Jobs were enqueued when an inspection dive was submitted, then processed in the background using FFmpeg and ImageMagick. The UI showed job status in real time. Report generation efficiency improved by 70%. More importantly, the system didn't block, didn't crash, and didn't lose jobs if something failed partway through.

What I Actually Learned

I learned Git properly here — not just "git commit" and "git push", but what remote and origin actually mean, how to stash and revert, how branching works on a shared codebase where someone else is also pushing. I broke things and fixed them. I read error logs. I learned to read Sequelize docs and PostgreSQL query plans.

The most valuable thing wasn't any specific technology. It was the habit of reading the whole codebase before touching it. The discipline of writing testable functions. The instinct to ask "what happens when this fails?" before shipping something. Those habits don't come from tutorials. They come from building something real for people who depend on it.
